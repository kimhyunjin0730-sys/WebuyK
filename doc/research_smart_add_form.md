# Research — 스마트 추가 단일 폼 (Smart Add Form)

> 작성일: 2026-04-09
> 대상 작업: We buy K 장바구니 추가 흐름의 통합/우아한 폴백
> 관련 사용자 요구: "어떤 사이트든 인식하게끔 하는 게 좋을 것 같은데 만약 사이트가 봇을 차단한다면 url을 가져오는 식이 아니라 링크 타고 가는 방식이 나으려나"

---

## 1. Goal

상품 URL을 붙여 넣는 사용자가 **어떤 사이트든** 좌절 없이 장바구니 추가까지 도달하게 만든다. 핵심 전환:

- **현재**: URL 폼이 막히면 → 빨간 에러 → 사용자 좌절 → 북마클릿이 별도 페이지에 있어서 발견 못 함
- **목표**: URL 폼이 막히면 → **자동으로 다음 옵션 안내** → 사용자가 한 번도 막히지 않음

---

## 2. Current state (검증된 사실)

### 2.1 디렉토리 / 주요 파일

| 파일 | 역할 | 핵심 사실 |
|---|---|---|
| `apps/web/src/app/[locale]/order/page.tsx` | URL 입력 폼 | 클라이언트 사이드 `detectVendor()`로 BLOCKED 사이트 즉시 감지 (line 38–55). `BLOCKED = [coupang, gmarket]`, `SUPPORTED = [Naver, Kream, 11st]` |
| `apps/web/src/app/[locale]/cart/page.tsx` | 장바구니 표시 + 빈 상태 | 빈 상태일 때 두 카드(URL 페이지로 / 북마클릿 페이지로) 표시 |
| `apps/web/src/app/[locale]/bookmarklet/page.tsx` | 북마클릿 설치 안내 | 드래그 가능한 북마클릿 (1.5KB inline JS), 헤더 nav에 골드 pill로 노출 |
| `apps/web/src/app/[locale]/bookmarklet/add/page.tsx` | 북마클릿 수신 페이지 | URL params(title/price/image/url)을 받아 `POST /cart/parsed` 호출 |
| `apps/api/src/products/product-parser.ts` | 백엔드 URL 파서 | `BLOCKED_VENDORS = [coupang, gmarket]`, fetch + cheerio 기반 JSON-LD/OG 추출 |
| `apps/api/src/cart/cart.controller.ts` | 카트 엔드포인트 | `POST /cart` (URL 기반, 백엔드 파싱), `POST /cart/parsed` (북마클릿 사전 파싱) |
| `apps/api/src/cart/cart.service.ts` | 카트 서비스 | `addByUrl()`, `addParsed()` 두 메서드 존재 |
| `apps/api/src/products/products.service.ts` | 제품 ingest 로직 | `ingestByUrl()` (1시간 캐시), `upsertParsed()` (북마클릿용) |

### 2.2 차단 사이트 사실 (실제 curl 검증)

| 사이트 | HTTP | 비고 |
|---|---|---|
| 쿠팡 | 403 (Akamai `<TITLE>Access Denied</TITLE>`) | 차단 확정 |
| G마켓 | 403 | 차단 확정 |
| 11번가 | 200 | 통과 (이번 세션에서 검증) |
| 네이버 스마트스토어 | 200 (예상) | 미검증 |
| Kream | 200 (예상) | 미검증 |

→ **확실한 차단**: Coupang, G마켓
→ **확실한 통과**: 11st (HTTP 200) — JSON-LD 유무는 별도 확인 필요
→ **미검증**: Naver, Kream — 표준 OG/JSON-LD 사용한다는 통념에 의존

### 2.3 현재 사용자 흐름 (지금 동작하는 방식)

```
URL 붙여넣기
   ├── 차단 사이트(쿠팡/G마켓) → 클라이언트에서 즉시 노란 경고 + 북마클릿 링크
   ├── 지원 사이트(Naver/Kream/11st) → 백엔드 fetch → JSON-LD/OG 추출
   │      ├── 성공 → 장바구니 (toast)
   │      └── 실패 → 빨간 에러 박스 + 토스트  ← 막다른 길
   └── 미지원 사이트 → "지원하지 않는 사이트" 에러  ← 막다른 길
```

**막다른 길 두 곳**: (a) 지원 사이트인데 파싱 실패, (b) 완전 미지원 사이트.
사용자는 이 두 경우에서 좌절하고 떠난다.

### 2.4 기존 엔드포인트 (재사용 대상)

| 엔드포인트 | Body | 동작 |
|---|---|---|
| `POST /cart` | `{sourceUrl, quantity, optionsNote?}` | 백엔드가 fetch → 파싱 → 장바구니 |
| `POST /cart/parsed` | `{sourceUrl, title, priceKrw, imageUrl?, vendor?, quantity, optionsNote?}` | 사전 파싱된 데이터로 직접 upsert (북마클릿 경로) |
| `POST /products/parse` | `{url}` | URL preview만 (장바구니 안 담음) |

→ **`POST /cart/parsed`가 이미 존재**한다. 수동 입력 폼이 만들어지면 이 엔드포인트를 그대로 재사용 가능 — 신규 백엔드 작업 0.

### 2.5 i18n 메시지 키

`apps/web/messages/{ko,en}.json`의 `order` 섹션:
- `order.url`, `order.qty`, `order.preview`, `order.addToCart`
- `order.coupangNotice` / `coupangNoticeLink` / `coupangNoticeTail` (이미 일반화돼서 "은(는) 봇 차단..."으로 시작)
- `order.supportedTitle`, `order.tip`

→ 기존 키는 거의 재사용 가능. "수동 입력" / "외부 페이지 열기" 같은 새 키만 추가.

## 3. Pain points (검증된 것)

1. **PP1 — 막힘이 곧 종료**: 파싱 실패 시 빨간 에러가 뜨고 그 자리에서 사용자가 할 수 있는 게 없음. "다음에 뭘 하지?"가 안 보임.
2. **PP2 — 북마클릿 발견성 부족**: 헤더의 골드 pill로 노출하긴 했지만, URL 폼에서 막힌 사용자가 헤더로 시선을 옮기는 행동은 자연스럽지 않음. **막힌 자리에 즉시 안내**가 필요.
3. **PP3 — "수동 입력" 출구 없음**: 사용자가 어떤 이유로든 자동 파싱을 못 쓰는 경우(파서가 망가짐, 사이트가 처음 보는 곳, 가격을 직접 보정하고 싶음), 우회로가 전혀 없음.
4. **PP4 — 두 갈래 흐름의 인지 부담**: "URL 페이지" + "북마클릿 페이지" 두 곳을 사용자가 외워야 함. 단일 진입점으로 통합되면 인지 부담 0.

## 4. 사용자가 제안한 방향 해석

> "url을 가져오는 식이 아니라 링크 타고 가는 방식이 나으려나"

두 가지 가능 해석:

### 해석 A — "사용자가 그 페이지로 직접 이동해서 처리"
이미 우리에겐 답이 있다: **북마클릿**. 사용자의 실제 브라우저에서 페이지를 열고 DOM에서 직접 추출. 봇 차단 100% 우회.

### 해석 B — "백엔드 fetch가 아닌 다른 가져오기 방식"
- 헤드리스 브라우저(Playwright + stealth) — 무한 cat-and-mouse, Coupang은 결국 막힘. **비추**
- 3rd-party 스크래핑 API(ScraperAPI 등) — 월 비용, 안정적. **운영 시 검토**
- 사용자 수동 입력 — 빠른 폴백. **본 작업의 핵심 폴백 후보**
- 비동기 큐(오퍼레이터가 사람 수동 처리) — 비즈니스 모델 변경. **본 작업 범위 외**

### 본 작업이 채택할 해석
**해석 A + 해석 B의 수동 입력 폴백** 을 결합한다. 이유:
- A 단독 — 북마클릿이 메인이 됨. 첫 사용자가 설치 안 한 경우 여전히 막힘.
- B 수동 입력 단독 — 사용자가 거부한 적 있음 ("수동입력이 되면 안되지").
- **A + B 조합** — 자동을 1순위로, 막히면 (a) "외부 페이지 열기 + 북마클릿 안내" 와 (b) "수동 입력" 두 개를 한 모달로 제시. 사용자는 항상 다음 액션이 보임. 수동 입력은 fallback이지 primary가 아니므로 사용자의 원래 우려도 지킴.

## 5. 제약 / 결정 사항

| 항목 | 결정 | 근거 |
|---|---|---|
| 백엔드 신규 엔드포인트 | **추가 안 함** | `POST /cart/parsed`가 이미 수동 입력에 충분 |
| `POST /cart` 보존 | **유지** | 자동 파싱 경로는 그대로. 새 흐름은 클라이언트에서 분기해서 두 엔드포인트 중 하나 호출 |
| 신규 페이지 추가 | **안 함** | 기존 `/order` 페이지를 리팩토링. 페이지 수가 늘면 사용자 인지 부담 증가 |
| 모달 vs 인라인 확장 | **인라인 확장** | 모달은 모바일 UX 나쁨, 인라인이 자연스럽고 키보드 흐름 좋음 |
| 새 사이트 차단 발견 시 대응 | `BLOCKED_VENDORS` 한 곳만 추가하면 양쪽(클라이언트 + 백엔드) 동기화 | parser와 order page 양쪽에 같은 리스트가 있는데, 향후 단일 소스로 통합하는 건 본 작업 범위 외 |
| `any` / `unknown` 사용 | **금지** | 개발 원칙. 이미 코드베이스가 깨끗함 |
| 신규 cursor pagination | **해당 없음** | 본 작업에 페이지네이션 자체가 없음 |

## 6. 알려진 한계 / 본 작업에서 다루지 않는 것

- **상품 이미지**: 수동 입력 폴백에선 사용자가 이미지 URL을 직접 넣어야 함 (자동 추출 불가)
- **가격 보정**: 자동 파싱 결과의 가격이 틀린 경우, 현재 흐름에선 수정 못 함. 본 작업에서 같이 해결 가능 (자동 파싱 후에도 결과를 편집 가능하게)
- **외부 페이지 열기 후 자동 복귀**: 사용자가 외부 페이지에서 북마클릿을 누르면 결국 우리 시스템으로 돌아오긴 하지만, 그 시점에 원래 `/order` 페이지의 입력 상태는 잃음. 별도 처리 필요 없음 (장바구니로 직행)

## 7. Open questions

1. **수동 입력 폴백을 모든 케이스에서 노출할지, "차단/실패"인 경우에만 노출할지** → 권장: 차단/실패에만. 자동 파싱이 1순위라는 시그널 유지.
2. **자동 파싱 성공 시 결과를 편집 가능하게 할지** → 권장: 가능하게. 사용자가 가격/제목 보정 가능하면 좋음. 단, "장바구니 담기" 버튼 옆에 "그대로 담기"와 "편집 후 담기" 두 액션이 있으면 인지 부담. 단일 폼으로 보여주고 모든 필드를 편집 가능하게 하되, 자동 파싱 결과는 placeholder처럼 prefill.
3. **차단 사이트 URL 입력 시 "외부 페이지 열기" 버튼을 제공할지** → 권장: 제공. `target="_blank"`로 새 탭에서 그 URL을 그대로 열어줌. 사용자가 그 페이지에서 북마클릿 클릭. 매우 자연스러움.
